**Status:** approved (auto-mode brainstorm)

**Authors:** Saksham + Claude Code (Opus 4.7)

## 1. Context

After SP1 (PR #354), SP2 (PR #359), and SP3 (PR #360), three of four sub-projects have shipped. SP4 was scoped with a "demand-gate first" warning because it duplicates capabilities OC already has (local Bash for code execution, multi-backend WebSearch). The user explicitly authorized building it now anyway.

**Honest framing:** SP4's value is narrow. It enables one thing OC genuinely cannot do locally: invoke Anthropic's pre-built `pptx`/`xlsx`/`docx`/`pdf` skills running in Anthropic's code-execution container. Without SP4, generating a PowerPoint via chat requires bundling python-pptx (and its libreoffice deps). With SP4, Anthropic's container does the work.

## 2. Goals

1. **Opt-in only.** No automatic inference, no automatic enable. User explicitly sets `runtime.custom["anthropic_skills"]` to enable.
2. **Single knob.** One field controls both beta header injection AND the `container.skills` request parameter AND the bundled `code_execution_20250825` tool (required by the API).
3. **Anthropic provider only.** Bedrock Converse doesn't expose this; OpenAI doesn't have it. Not a cross-provider abstraction.
4. **Honest documentation.** When to use, when NOT to (cloud round-trip latency; cost — server-side execution is metered; ZDR-not-eligible).

## 3. Non-goals

- **No server-side `web_search` tool wiring.** OC has its own multi-backend WebSearch (`opencomputer/tools/websearch.py`); duplicating server-side adds cloud dep without value.
- **No standalone `code_execution` tool.** Only enabled as a side-effect of `anthropic_skills` being non-empty (skills require it). If OC ever needs sandboxed code execution standalone (e.g., for headless mode), that's a separate feature.
- **No CLI command** like `oc skills enable anthropic:pdf`. The env var / runtime knob is enough; CLI surface is YAGNI.
- **No download path for model-created files.** That's SP3's `oc files download`. Already shipped.
- **No automatic inference** ("user asked for PowerPoint → enable pptx skill"). Heuristics are unreliable; explicit opt-in is honest.
- **No `Skills` API client wrapper** (list/get/upload custom skills). Out of scope; only consumes Anthropic-managed skills.
- **No schema for response files.** Files generated by skills are accessible via existing `oc files download` (SP3) on the API side; user retrieval pattern unchanged.

## 4. Approach

A small adapter on the Anthropic provider's request-build path. When `runtime.custom["anthropic_skills"]` is present and non-empty:

1. Inject beta headers: `code-execution-2025-08-25`, `skills-2025-10-02` (and `files-api-2025-04-14` for skill-output retrieval).
2. Add `container.skills` array to the request body, listing each skill_id with `type=anthropic, version=latest`.
3. Add `code_execution_20250825` tool to the `tools` array (required for skills to actually run).

Provider-level. No agent-loop changes. No new public API on `BaseProvider`.

The runtime knob is a list of skill IDs:

```python
runtime.custom["anthropic_skills"] = ["pdf", "pptx", "xlsx", "docx"]
```

Empty/missing → no beta headers, no container.skills, no extra tool. Today's behavior unchanged.

## 5. Design

### 5.1 Runtime flag definition

`runtime.custom["anthropic_skills"]: list[str]` — Anthropic-managed skill IDs. Validated only at use-site (provider checks types defensively; bad input → log warning + ignore).

Configuration paths (any one works):
- Programmatic: agent caller sets `runtime.custom["anthropic_skills"] = [...]`
- Environment (NEW): `OPENCOMPUTER_ANTHROPIC_SKILLS=pdf,pptx` — comma-separated; parsed by the same code path that resolves model/provider/etc. env vars

Slash command? Defer. If usage emerges, add `/anthropic-skills pdf,pptx` in a follow-up.

### 5.2 Provider integration ([`extensions/anthropic-provider/provider.py`](../../../extensions/anthropic-provider/provider.py))

A new helper:

```python
ANTHROPIC_SKILLS_BETA_HEADERS = (
    "code-execution-2025-08-25",
    "skills-2025-10-02",
    "files-api-2025-04-14",
)
CODE_EXECUTION_TOOL = {
    "type": "code_execution_20250825",
    "name": "code_execution",
}


def _resolve_anthropic_skills(runtime) -> list[str]:
    """Get the list of Anthropic skill IDs to enable for this call.

    Resolution order:
    1. runtime.custom["anthropic_skills"] (explicit programmatic)
    2. OPENCOMPUTER_ANTHROPIC_SKILLS env var (comma-separated)
    3. [] (no skills)

    Bad input (non-list, non-strings) is logged and ignored.
    """
    if runtime is not None:
        explicit = (getattr(runtime, "custom", {}) or {}).get("anthropic_skills")
        if explicit is not None:
            if isinstance(explicit, list) and all(isinstance(s, str) for s in explicit):
                return [s for s in explicit if s.strip()]
            log.warning(
                "anthropic_skills runtime flag has bad type %r; expected list[str]",
                type(explicit).__name__,
            )
            return []
    env = os.environ.get("OPENCOMPUTER_ANTHROPIC_SKILLS", "").strip()
    if env:
        return [s.strip() for s in env.split(",") if s.strip()]
    return []


def _build_skills_container(skill_ids: list[str]) -> dict:
    """Build the container.skills array per Anthropic API spec."""
    return {
        "skills": [
            {"type": "anthropic", "skill_id": sid, "version": "latest"}
            for sid in skill_ids
        ]
    }


def _augment_kwargs_for_skills(
    *,
    kwargs: dict,
    skill_ids: list[str],
) -> dict:
    """Mutate kwargs to enable Anthropic Skills-via-API.

    - Adds extra_headers with the beta headers (preserving any existing ones)
    - Adds container.skills
    - Appends code_execution_20250825 to tools (avoid duplicating if already present)
    """
    if not skill_ids:
        return kwargs

    # Beta headers
    extra = dict(kwargs.get("extra_headers") or {})
    existing_betas = [b.strip() for b in extra.get("anthropic-beta", "").split(",") if b.strip()]
    for beta in ANTHROPIC_SKILLS_BETA_HEADERS:
        if beta not in existing_betas:
            existing_betas.append(beta)
    if existing_betas:
        extra["anthropic-beta"] = ",".join(existing_betas)
    kwargs["extra_headers"] = extra

    # container.skills
    kwargs["container"] = _build_skills_container(skill_ids)

    # code_execution tool (required for skills to run)
    tools = list(kwargs.get("tools") or [])
    if not any(t.get("type") == "code_execution_20250825" for t in tools):
        tools.append(CODE_EXECUTION_TOOL)
    kwargs["tools"] = tools

    return kwargs
```

Call site: in `complete()` and `stream_complete()` just before `client.messages.create(**kwargs)` / `client.messages.stream(**kwargs)`, call `_augment_kwargs_for_skills(kwargs=kwargs, skill_ids=_resolve_anthropic_skills(runtime))`.

### 5.3 Behavior matrix

| Provider | Behavior |
|---|---|
| **Anthropic** (native + Bearer-proxy modes) | Honors `anthropic_skills`. Cloud round-trip per call; latency penalty; cost penalty (skills run server-side and bill compute time). |
| **AWS Bedrock** | Ignored. Bedrock doesn't support Anthropic Skills-via-API. Logs warning if user sets the flag. |
| **OpenAI** + OpenAI-compat | Ignored (no equivalent feature). Logs warning. |
| **All other providers** | Ignored (default behavior — `_resolve_anthropic_skills` is only called by Anthropic). |

### 5.4 Errors

- Bad type for `anthropic_skills` (e.g., `True`, `"pdf"` instead of `["pdf"]`) → warning + ignore.
- Empty list → no-op (not an error).
- Skill ID rejected by Anthropic API → 400 from server; the provider's existing error path surfaces it. We don't pre-validate skill IDs (they could change; let the API be the source of truth).

## 6. Decisions log

| Decision | Why |
|---|---|
| Single runtime flag (`anthropic_skills` list) | One knob is easier to reason about than separate flags per skill |
| No automatic skill-ID inference | Heuristics are unreliable; explicit opt-in is honest |
| No standalone `code_execution` flag | Bundled with skills (which require it); separate use case is YAGNI |
| Skip server-side `web_search` | OC's multi-backend WebSearch is better; cloud round-trip adds nothing |
| Bedrock + OpenAI: log + ignore | Cross-provider portability is the SDK's job; this feature is Anthropic-specific |
| No CLI command | Env var is enough for the dogfood phase |
| `version: "latest"` (not pinned) | Anthropic owns these skills; we don't curate versions; if a user needs pinning, they can set `runtime.custom` programmatically with full skill dicts |

## 7. Risks

1. **Beta-header churn.** All three beta headers may be promoted to GA or rotated. Mitigation: constants in one place; one-line update.
2. **User accidentally enables skills with no understanding.** Each call now incurs a cloud round-trip + server-side compute. Mitigation: documented prominently; default OFF.
3. **Skill ID typos.** Pre-validation deferred (Anthropic owns the canonical list and may add skills). User gets a 400 from the server; provider's existing error path surfaces it.
4. **Cost surprise.** Server-side skill execution is metered. We don't add cost guards in this PR (out of scope; OC's existing cost-guard hooks can be wired in a follow-up if usage is real).

## 10. Out of scope (final list — nothing left after SP4)

- Server-side `web_search` (OC has multi-backend equivalent)
- Standalone `code_execution` flag (bundled with skills)
- CLI command (`oc skills enable anthropic:pdf`) — env var is sufficient
- Custom user-uploaded Skills-API workflow (consume only, don't publish)
- Cost guards on server-side skill execution (separate follow-up if usage emerges)
- Auto-inference of skill IDs from prompt content (heuristics unreliable)
- Bedrock / OpenAI Skills-API support (no API equivalent exists)
