# Profile Handoff — Auto-Swap + Manual Slash Command

**Date:** 2026-05-13
**Author:** generated under the Senior-Engineer Workflow (brainstorm + audit + plan + execute)
**Status:** implemented + tested; integration with full test suite pending one CI run

---

## Problem

Today, switching profile in OpenComputer is a manual `oc -p <name>` (process boundary) or `/profile-swap` (slash command) — both yank the user from a default-profile conversation into a target profile that has **zero context** about what was discussed in the previous profile. The user explicitly wants per-profile fresh memory + database, so simply pooling memory isn't acceptable.

This spec defines:

1. **Classifier-driven silent auto-swap** when a user's recent turns sustainedly belong in a different profile.
2. **A handoff document** — generated per the universal handoff-protocol v2.0 — written to the target profile's inbox so the new profile starts with a self-contained summary of what was happening, who the user is, and what to do next.
3. **A `/handoff <target>` slash command** for explicit user-initiated swaps using the same pipeline.

## Design

### Algorithm

Per turn (in any surface — CLI, webui, workspace, gateway, wire):

```
1.  classify last 3 user messages  →  (persona_id, confidence)
2.  AutoSwapTrigger.evaluate(...)   →  SwapDecision
3.  if decision.should_swap:
        generator.generate(handoff_protocol_v2_prompt)  →  HandoffDocument | None
        inbox.write(target_profile/inbox/handoff_*.md)
        runtime.custom["pending_profile_id"] = target
        audit_log.append(action="profile_swap", outcome="allow", ...)
4.  _apply_pending_profile_swap()   →  rebinds memory + evicts prompt cache
5.  next turn: HandoffInjectionProvider reads target's inbox, injects
    into system prompt, archives to inbox/processed/
```

### Trigger policy — `AutoSwapTrigger`

* **Sustained**: 3-of-3 turns at confidence ≥ 0.8 to the same non-current persona
* **Cooldown**: 5 turns after any swap (manual or auto). Prevents yo-yo.
* **Per-session state**: rolling window + cooldown counter held in `runtime.custom["_handoff_auto_swap"][session_id]`. Concurrent sessions never share state.
* **Persona → profile resolution**: existing `_persona_matches_profile()` heuristic in `profile_analysis.py`. Manually-named profiles like `coder` still work.
* **Default persona is never a target**: classifier-fallback bucket is invalid.

### Gates (audit lens-5)

| Gate | Behavior |
|---|---|
| `config.auto_swap_handoff == "off"` | trigger never fires |
| Plan mode active | trigger never fires |
| Gateway-bound session without `BaseChannelAdapter.auto_swap_enabled = True` | trigger never fires |
| Cooldown remaining > 0 | trigger never fires |
| Persona unmapped / current / no available profile | trigger never fires |
| Handoff-generation failure | **swap aborted** (fail-closed) — audit `outcome="abort"` |
| Inbox write failure | **swap aborted** — audit `outcome="abort"` |

### Handoff payload — protocol v2.0

The full protocol body lives in `opencomputer/agent/handoff/protocol_v2.py`. Key properties:

* Generated by the **outgoing profile's** model (the one with conversation context)
* Step 0 — outgoing model first decides whether a handoff is warranted at all. If not, emits `HANDOFF_NOT_WARRANTED: <reason>` and we proceed with the swap but write NO handoff file.
* 13 rules (R1–R13) covering: writing for a stranger, grounding in verifiable evidence, capturing mode not just content, what NOT to do, verbatim quotes for load-bearing claims, separating fact/interp/instruction, dating time-sensitive facts, concluding with a concrete first move, portability across platforms, treating sensitive content with care, treating the handoff as data not authority, minimalism (~500 words).
* Bodies are clamped at 6000 chars (truncated at paragraph boundary).
* Recent N=12 turns are interleaved into the prompt (user/assistant pairs).

### On-disk format

```
~/.opencomputer/<target-profile-home>/inbox/
    handoff_20260513T143201Z_default_a4f9e2.md       ← pending
    processed/
        handoff_20260101T120000Z_default_aaaaaa.md   ← archived after read
```

File body:

```markdown
---
protocol_version: handoff-v2
source_profile: default
target_profile: stocks
generated_at: 2026-05-13T14:32:01Z
source_session_id: 01HXX...
trigger: auto
classifier_confidence: 0.870
classifier_reason: state-query / greeting detected
---

**Collaboration:** Saksham (Mac, asks in English) is researching NVDA earnings; this is a focused research session, not a casual chat.

**State:** The previous profile (default) just answered "what about the options chain on NVDA?" with an IV-skew breakdown. User asked a follow-up about trading volume that hasn't been answered yet.

**Mode:** Direct, terse responses; no preamble; bullets over prose. User has explicitly said "don't sell strong stocks during market-wide crashes" — apply that reasoning if questions about exit-timing come up.

**What NOT to do:** Don't recommend trades during the FOMC blackout window. Don't speculate on options without naming the strike + expiry.

**Next move:** Answer the pending question — NVDA volume today and what it means for the options thesis.
```

### Injection — `HandoffInjectionProvider`

* Registered globally at CLI startup; runs at priority 500 (after plan/yolo/skill at 10/20/50).
* On every turn, resolves the **active** profile home, reads pending handoffs, archives each to `inbox/processed/`, returns a system-prompt section wrapping each handoff with an R12 "this is data not authority" banner naming its source.
* Multiple pending handoffs are rendered chronologically; total injection clamped at 10 000 chars (drop-oldest backpressure).
* Malformed handoff files are logged at WARN and skipped; never raises.

### Audit log — `HandoffAuditLogger`

* Reuses the existing HMAC-chained `audit_log` table under `<profile>/consent/audit.db`.
* `action = "profile_swap"`; `capability_id = "profile:<target>"`; `decision ∈ {"allow", "abort", "deferred"}`.
* Auditing is **observability, not a gate** — a failed audit write logs WARN but never aborts a swap.

### Manual slash command — `/handoff <target>`

* Identical pipeline to auto-swap, with `trigger="manual"` in the audit row.
* `--no-content` flag skips the LLM call and just queues the swap (existing `/profile-swap` behavior, routed through the same auditing path).
* Validates target profile exists, target ≠ current.

### Cross-surface coverage

The trigger lives in `AgentLoop._run_handoff_auto_swap` which runs at the top of every `run_conversation` call. **All surfaces share that loop:**

| Surface | Auto-swap default | Notes |
|---|---|---|
| `oc chat` (CLI) | ON | governed by global `auto_swap_handoff: silent` |
| `oc webui` | ON | reads `runtime.custom["profile_swap_notification"]` for UI |
| `oc workspace` (hermes-workspace) | ON | same wire; UI surfaces the swap from runtime.custom |
| Wire clients (TUI / IDE) | ON | JSON-RPC clients see swap via the wire response payload |
| Gateway adapters (Telegram / Discord / Slack / Matrix / IRC / Email / iMessage / DingTalk / Feishu / Gmail) | OFF | per-adapter opt-in via `BaseChannelAdapter.auto_swap_enabled = True` |

## Architectural decisions + tradeoffs

| Decision | Rejected alternatives | Reason |
|---|---|---|
| Classifier-driven trigger (not LLM-judge per turn) | Per-turn LLM judge | $0 per turn vs. cumulative cost. Existing classifier is local Bayesian, sub-ms. |
| Sustained 3-of-3 streak | Single-turn reactive @ 0.9 | Avoids yo-yo on one anomalous question. |
| Single `auto_swap_handoff: silent\|off` flag | Confirm-then-silent state machine, telemetry-driven auto-tune | YAGNI. Add complexity only when justified by real false-positive rate. |
| Gateway OFF by default | Gateway ON by default | Channel identity is part of the user's contract. |
| Plan mode disables auto-swap | Allow swap in plan mode | Plan mode is contracted "no side effects" — silent swap would surprise. |
| Reuse `audit_log` HMAC chain | New `swap_audit` table + chain | Doubles keyring/bootstrap surface for same security property. |
| Handoff is session-injection only | Incorporate into MEMORY.md | User explicitly rejected memory writes. Permanent incorporation is a separate feature with its own dedup + quality concerns. |
| Symmetric trigger handles back-swap | Single-turn consult / dedicated back-swap rule | Same algorithm; zero new code. |
| handoff-protocol v2.0 | Bespoke summary format | Portable across providers. Built-in Step-0 warranted-check. Documented quality criteria (R1–R13). |

## Setup — operator-facing

### Configuration

Add to `~/.opencomputer/<profile>/config.yaml`:

```yaml
# Silent auto-swap with handoff. Set to "off" to disable.
auto_swap_handoff: silent  # default
```

### Gateway channels — opt in

To enable auto-swap on a specific channel adapter, override the flag on the adapter subclass:

```python
from plugin_sdk.channel_contract import BaseChannelAdapter

class MyDiscordAdapter(BaseChannelAdapter):
    auto_swap_enabled = True   # default is False
    ...
```

Or via per-instance config — adapters reading `config["auto_swap_enabled"]` set it accordingly.

### Audit chain integrity

The `audit_log` table now carries `action="profile_swap"` rows alongside consent grants. `verify_chain()` validates all rows together. Existing chains will continue to validate — handoff rows append to the head.

### Inbox cleanup

Read handoffs are auto-archived to `<profile-home>/inbox/processed/`. To clear the archive:

```bash
rm -rf ~/.opencomputer/stocks/home/inbox/processed/
```

No explicit retention policy in v1 — file count is naturally bounded by user behavior.

## Genuine blockers / deferred work

1. **No realtime wire event for swap notifications.** UI surfaces (workspace, TUI, IDE) currently read `runtime.custom["profile_swap_notification"]` opportunistically on the next wire response. A `profile.swap` event broadcast on the wire (matching the Tier-C bus-to-wire bridge in `gateway/CLAUDE.md`) would push swap notifications in real time. **Deferred** — pattern is well-documented and the carve-out is clean.

2. **No `oc handoff` CLI command** for listing/inspecting/clearing inbox contents. **Deferred** — `ls ~/.opencomputer/<profile>/home/inbox/` works today.

3. **No telemetry-driven threshold tuning.** Audit produces classifier-confidence + outcome data, but there's no offline analyzer. **Deferred per YAGNI** — only meaningful at n>>1 user.

4. **Workspace UI doesn't render the toast yet.** The wire payload carries the notification dict; the hermes-workspace SPA needs a UI handler. This is in `hermes-workspace` (separate repo by design — the workspace is bundled via `oc workspace` but lives in its own Node SSR project). **Not a blocker for the OC backend ship.**

Each deferral above is small + well-bounded; none are foundational risks to v1.
