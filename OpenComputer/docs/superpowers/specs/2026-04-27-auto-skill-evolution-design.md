# Auto Skill Evolution — Self-Improving Skills from Successful Runs

**Date:** 2026-04-27
**Status:** Design
**Branch:** `feat/auto-skill-evolution`
**Worktree:** `/tmp/oc-skill-evo/`

---

## 1. Goal

Close OC's only real gap vs. Hermes's "self-improving / agent-created skills" claim: an **automatic** loop that watches successful sessions, detects reusable patterns, generates skill candidates, and stages them for user review (default) or auto-publishes (opt-in). The user retains control; the agent does the noticing + drafting.

## 2. Why this and not "auto-publish on success"

The naive design — "session ends, save the whole thing as a skill" — pollutes the skills directory within hours. After a week of dogfood you'd have 200 one-off "skills" describing single sessions. The right shape is:

1. **Detection**: filter the noise. Most sessions are NOT reusable patterns; only a few are.
2. **Extraction**: convert a session into a generalized SKILL.md, not a verbatim transcript.
3. **Quality gate**: dedup against existing skills; reject low-confidence drafts.
4. **Staging**: candidates land in `<profile_home>/skills/_proposed/` and need explicit user accept (default) OR auto-promote with confidence threshold (opt-in).

This matches how the broader memory system already works (V2.B deepening loop runs in background, surfaces signals; doesn't auto-act).

## 3. Architecture

### 3.1 Module shape

```
extensions/skill-evolution/
├── plugin.json              # manifest (kind=mixed, default OFF)
├── plugin.py                # registration + start subscriber if state.enabled
├── pattern_detector.py      # heuristic + LLM judge for "reusable pattern"
├── skill_extractor.py       # session → SKILL.md draft
├── candidate_store.py       # stage / load / accept / reject candidates
├── subscriber.py            # F2 bus listener for SessionEndEvent + ToolCallEvent
├── README.md                # privacy contract + usage
```

### 3.2 New SDK event type — `SessionEndEvent`

Currently F2 has `TurnStartEvent` but no `SessionEndEvent`. Add to `plugin_sdk/ingestion.py`:

```python
@dataclass(frozen=True, slots=True)
class SessionEndEvent(SignalEvent):
    """Session reached terminal state (END_TURN, error, or user-closed)."""

    event_type: str = "session_end"
    end_reason: str = "completed"      # "completed" | "error" | "cancelled" | "timeout"
    turn_count: int = 0
    duration_seconds: float = 0.0
    had_errors: bool = False
```

Emitted by the agent loop at the same point title generation fires (TS-T6 from prior work). Useful for any future subscriber (analytics, evolution, etc.) — not just this feature.

### 3.3 Pattern detector (heuristic + LLM judge)

Two-stage filter:

**Stage 1 — heuristic filter** (cheap, runs every session end):
- Skip sessions with `had_errors=True`.
- Skip sessions with turn_count < 3 (too short to be a "pattern").
- Skip sessions matching active skills (FTS5 search on session content vs existing skill descriptions).
- Skip sessions whose user messages are conversational ("hi", "thanks", weather questions).

Sessions passing stage 1 are ~10% of total.

**Stage 2 — LLM judge** (expensive, only on stage-1 passes):
- Cheap model (claude-haiku) given a structured prompt: session transcript summary + existing skills + "is this a NOVEL reusable pattern? confidence 0-100, one-sentence why".
- Threshold: confidence ≥ 70 advances to extraction.
- Cost-guarded: max N judge calls per day (configurable, default 20).

### 3.4 Skill extractor

Given a passing session, generate a SKILL.md candidate:
1. Extract the user's intent (LLM call: "summarize what the user was trying to accomplish in 1 sentence").
2. Extract the procedure (LLM call: "summarize the agent's successful procedure as numbered steps").
3. Generate trigger description (LLM call: "what user request would benefit from this pattern? phrase as a 'use when X' description").
4. Compose SKILL.md with frontmatter + body in OC's standard format.
5. Save to `<profile_home>/skills/_proposed/<auto-generated-name>/SKILL.md`.

The generated skill name is `auto-{session-id-prefix}-{slug-from-intent}` so it's clearly auto-generated and traceable.

### 3.5 Candidate store

Files at `<profile_home>/skills/_proposed/<name>/`:
- `SKILL.md` — the proposed skill
- `provenance.json` — metadata: session_id, generated_at, confidence_score, source_summary

Operations:
- `add_candidate(skill_md, provenance)` — atomic write
- `list_candidates()` — return all proposed skills with metadata
- `accept_candidate(name)` — moves to `<profile_home>/skills/<name>/` (active)
- `reject_candidate(name)` — deletes the proposal
- `get_candidate(name)` — load full SKILL.md + provenance for review

### 3.6 CLI: `oc skills`

New subcommand group on the main Typer app:

```bash
oc skills list                  # all skills (active + proposed) with provenance
oc skills review                # interactive review of proposed candidates
oc skills accept <name>         # promote a candidate to active
oc skills reject <name>         # delete a candidate
oc skills evolution {on,off}    # toggle the auto-evolution subscriber
oc skills evolution status      # show recent activity (proposed count, accept rate)
```

`oc skills evolution status` shows aggregate-only (proposed count, accept rate, last session analyzed). Never specific session content.

### 3.7 F1 capability claim

```python
"skill_evolution.observe": ConsentTier.IMPLICIT     # subscribe to bus
"skill_evolution.propose": ConsentTier.EXPLICIT     # generate candidate (LLM call)
"skill_evolution.auto_publish": ConsentTier.PER_ACTION  # auto-promote without review
```

Three tiers because the activities have different risk profiles:
- Observing the bus: zero risk → IMPLICIT
- Generating candidates: costs LLM calls → EXPLICIT
- Auto-publishing without review: alters the active-skills directory → PER_ACTION (every promotion needs explicit grant or per-promotion confirmation)

### 3.8 Default policy

| Setting | Default | Behavior |
|---|---|---|
| Auto-evolution subscriber | OFF | User opts in via `oc skills evolution on` |
| Auto-publish threshold | None (review-only) | Candidates ALWAYS go to `_proposed/`; user runs `oc skills review` |
| Daily LLM judge budget | 20 calls/day | Cost-guard prevents runaway spend |
| Privacy redaction | ON | Sensitive content (paths matching `*api_key*`, paths under `~/Documents/Banking`, etc.) excluded from extraction |

Auto-publish (without review) is opt-in via `oc skills evolution config --auto-publish-confidence 90` — and even then the consent gate trips PER_ACTION on each promotion. User can revoke at any time.

### 3.9 Privacy contract (hard, baked into tests)

The skill-evolution subscriber must NOT:
- ❌ Send any session content to a network destination beyond the configured LLM provider for the judge/extract calls.
- ❌ Include sensitive-app content (per the ambient-sensor sensitive deny-list) in extraction.
- ❌ Persist raw session transcripts in `_proposed/` — only the GENERATED SKILL.md + minimal provenance.
- ❌ Auto-promote without an explicit policy flag.
- ❌ Run when paused or disabled.
- ❌ Share user data with other users or external systems.

Tests:
- `test_skill_evolution_no_raw_transcript_persisted.py` — proposes a candidate, verifies no transcript file in `_proposed/`.
- `test_skill_evolution_sensitive_apps_excluded.py` — session with sensitive-app foreground events doesn't generate a candidate.
- `test_skill_evolution_cost_guard.py` — exceeds daily judge budget → no more judge calls.
- `test_skill_evolution_privacy_redaction.py` — paths/strings matching deny-list never appear in candidate SKILL.md.

## 4. Integration with existing OC infra

This feature reuses heavily — almost no new infrastructure:

| Need | Reuses |
|---|---|
| Subscribe to session events | F2 TypedEventBus (`opencomputer/ingestion/bus.py`) |
| Run LLM judge / extractor | Existing provider plugins (anthropic-provider) via `PluginRegistry.providers[...]` |
| Cost guard | `opencomputer/cost_guard.py` (already present, used by voice tools) |
| Detached background work | `SpawnDetachedTaskTool` (T5 from prior session) — proposal task runs async |
| Capability gating | F1 ConsentGate (3 capability claims declared) |
| Sensitive-app filter | Reuse `extensions/ambient-sensors/sensitive_apps.py::is_sensitive` for redaction |
| Skill loading | OC's existing skill loader scans `<profile_home>/skills/` and `_proposed/` is filtered out (only active skills load by default) |
| CLI mount | `opencomputer/cli.py` Typer app — same pattern as `ambient`, `voice`, etc. |

The only NEW core types are `SessionEndEvent` (in `plugin_sdk/ingestion.py`) and the 3 capability claims.

## 5. Risks + mitigations

| Risk | Mitigation |
|---|---|
| LLM costs spiral if user has many sessions | Daily budget (`max_judge_calls_per_day`, default 20); cost-guard pre-flight check |
| LLM judge marks every session "novel pattern" → spam in `_proposed/` | Confidence threshold (default 70); dedup against existing skill descriptions via FTS5 |
| Sensitive content leaks into `_proposed/` SKILL.md | Reuse ambient-sensor sensitive-app filter on session content before extraction; redact path strings; no raw transcripts persisted |
| User accumulates 100+ proposals and never reviews | `oc skills evolution status` shows proposed count; warn at >20 unreviewed; auto-prune proposals older than 90 days |
| Auto-published skills override user's curated skills | Generated names always prefix `auto-` — never collide with curated skill names |
| Skill quality degrades over time | Each proposal stores confidence + provenance; user can sort review queue by confidence |
| Subscriber crashes silently | Heartbeat file at `<profile_home>/skills/evolution_heartbeat`; doctor warns if stale |
| User wants to disable mid-session | `oc skills evolution off` flips state; subscriber stops within one bus tick |

## 6. Out of scope (deferred)

- Cross-user skill sharing (federated skill marketplace) — Tier 4 vision
- LLM-graded skill QUALITY scoring (rubric-based) — V2 of this feature
- Auto-deletion of unused skills (last-used timestamp pruning) — separate plan
- Skill versioning / migration when skill schema changes — separate plan
- "Watch the skill being applied and verify it works" closed-loop — separate plan

## 7. Self-review

**Spec coverage**: each framework-lens default in §3.8 is implemented in the corresponding §3.x section. Privacy contract in §3.9 is enforceable via the listed tests.

**Internal consistency**: detector → extractor → store → CLI flow is single-direction. State file at `<profile_home>/skills/_proposed/` is the single source of truth. F1 capabilities are declared once and used everywhere via existing gate.

**Scope check**: 8 implementation tasks. Realistic at ~1,100 LOC + ~500 LOC tests = ~1,600 total. ~7-8 hours via subagent-driven dev with TDD.

**Ambiguity check**: "what counts as a successful session?" — defined in §3.3 stage-1 heuristic (no errors, ≥3 turns, not a duplicate of existing skill, non-conversational). Edge cases listed in §5.

**Framework-lens validation**: default OFF; opt-in via `oc skills evolution on`; staging directory keeps user in control; per-call cost guard; LLM-provider-agnostic (uses existing provider plugins). A random user installing OC sees no behavior change.
