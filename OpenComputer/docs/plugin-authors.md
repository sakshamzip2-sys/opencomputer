# Writing plugins for OpenComputer

This guide is the 30-minute path from zero to a loaded plugin. It is
deliberately focused — read it start-to-finish once, then use
[`sdk-reference.md`](./sdk-reference.md) as a lookup when you need the
shape of a specific type.

## 1. "Hello, plugin" in 60 seconds

Scaffold a new toolkit plugin and verify it loads:

```bash
opencomputer plugin new my-thing --kind toolkit
opencomputer plugins   # should list "my-thing" among the discovered plugins
```

That produces a working plugin tree under
`~/.opencomputer/plugins/my-thing/` containing `plugin.json`, `plugin.py`,
a `tools/` package with one starter tool, `tests/`, and a `README.md`.
The last line of the scaffolder output is `✓ Smoke check passed` — the
CLI actually imports the plugin through the real loader with an isolated
registry, so a template that produces broken code fails loudly here
rather than at agent start-up.

For a fully-worked provider example you can read, see
[`extensions/weather-example/`](../extensions/weather-example/) — a
bundled plugin generated by this same scaffolder.

## 2. The manifest (`plugin.json`)

Every plugin starts with a manifest. Fields come from
`plugin_sdk.core.PluginManifest`:

| Field | Type | Description & when to set |
|---|---|---|
| `id` | string | Stable id, lowercase letters/digits/hyphens, 1–64 chars. Used for install paths and collisions. Set once, never change. |
| `name` | string | Human-readable display name shown in `opencomputer plugins`. Free-form. |
| `version` | string | SemVer string. Bump on every release. |
| `description` | string | One-sentence blurb. Shown in lists. Defaults to `""`. |
| `author` | string | Free-form. Shown in README. Defaults to `""`. |
| `homepage` | string | URL; optional. |
| `license` | string | SPDX id. Defaults to `"MIT"`. |
| `kind` | enum | One of `channel`, `provider`, `tool`, `skill`, `mixed`. Determines which registry methods `register()` calls. |
| `entry` | string | Path to the entry module (without `.py`), relative to the plugin root. The loader imports this module and calls its `register(api)`. |
| `profiles` | `["*"]` or list of names | Phase 14.C profile scoping. `null`/`["*"]` = any profile. A list like `["coding", "research"]` restricts the plugin to those profiles only. Set when a plugin should only be visible to certain workspace profiles. |
| `single_instance` | bool | True if the plugin owns an exclusive resource (e.g. a bot token) and can only belong to ONE profile at a time. Enforced by an atomic PID lock at `~/.opencomputer/.locks/<plugin-id>.lock`; a second profile trying to load the plugin raises `SingleInstanceError`. Stale locks from crashed processes are stolen atomically on the next load. |
| `enabled_by_default` | bool | True if the plugin should be auto-enabled on a fresh install. Today only `memory-honcho` uses this. Leave `false` unless you're shipping a core overlay. |

Note: the CLI `--kind` flag uses the UX-friendly `toolkit`, but the
manifest records the SDK enum value `tool`. The template handles that
mapping; you never write `toolkit` in `plugin.json` by hand.

## 3. The four kinds

A plugin's `kind` field decides which registry it writes to in `register()`.

| Kind | Used for | Extend | Register via |
|---|---|---|---|
| `channel` | New messaging platform (Telegram, Slack, Matrix). Inbound events become `MessageEvent`; outbound calls hit `send()`. | `BaseChannelAdapter` | `api.register_channel(name, adapter)` |
| `provider` | New LLM backend (Anthropic, OpenAI, local llama.cpp). Agent loop is provider-agnostic; this plugs one in. | `BaseProvider` | `api.register_provider(name, provider)` |
| `tool` (CLI: `toolkit`) | New tool(s) the agent can call (Read, Write, Grep, custom APIs). | `BaseTool` per tool | `api.register_tool(tool)` — call once per tool |
| `mixed` | Plugins that contribute several of the above (e.g. a channel + its custom slash-commands). | Multiple base classes | Call each `register_*` method as needed |

Skills are a separate category — they live under
`~/.opencomputer/skills/` and ship as Markdown + frontmatter, not as
`kind=skill` plugins (that enum value exists for forward-compat but no
plugin currently uses it).

## 4. The `register(api)` contract

Every plugin's entry module (`entry` field in the manifest) exports a
top-level function:

```python
def register(api) -> None:  # PluginAPI is duck-typed
    api.register_provider("my-provider", MyProvider)
```

The loader imports the entry module, then calls `register(api)` with a
`PluginAPI` instance. `api` exposes only what plugins are allowed to
do — **never** reach into `opencomputer.*` from inside `register`.

Available `api` methods (authoritative list in
[`plugin_sdk/__init__.py`](../plugin_sdk/__init__.py)):

| Method | Purpose |
|---|---|
| `api.register_tool(tool: BaseTool)` | Add a tool the agent can call. |
| `api.register_provider(name: str, provider)` | Register a `BaseProvider` subclass *or instance* under a short name. |
| `api.register_channel(name: str, adapter)` | Register a `BaseChannelAdapter` subclass *or instance*. |
| `api.register_hook(spec: HookSpec)` | Register a lifecycle hook (PreToolUse, PostToolUse, etc.). |
| `api.register_injection_provider(provider: DynamicInjectionProvider)` | Add a system-prompt modifier (plan mode, yolo mode, custom modes). |
| `api.register_memory_provider(provider: MemoryProvider)` | Plug in an external memory backend. Only ONE allowed per session. |
| `api.register_doctor_contribution(contribution: HealthContribution)` | Add a `opencomputer doctor` check. |

`register` should be cheap. Do I/O, hit the network, or spawn
background tasks elsewhere — hooks, first tool call, connect() on the
channel adapter. A slow `register` slows every agent start.

## 5. Lifecycle

Plugin discovery is **two-phase**, modeled on OpenClaw:

1. **Discovery (cheap).** The CLI / agent scans every configured root
   (`extensions/`, `~/.opencomputer/plugins/`, profile-local dirs) for
   `plugin.json` files. Only JSON parsing happens — no `import`s, no
   side-effects. This is what `opencomputer plugins` uses.
2. **Loading (lazy).** When the agent actually needs a plugin (first
   turn, or on channel start for the gateway), the loader imports
   `entry` and calls `register(api)`. Entry modules are registered
   under synthetic names (`_opencomputer_plugin_<id>_<entry>`) to
   avoid `sys.modules` collisions between sibling `plugin.py` files.

**Teardown: there isn't one** (today). Plugins that allocate resources
in `register` (e.g. open WebSocket connections) leak them until process
exit. Channel adapters have `connect()`/`disconnect()` managed by the
gateway, but tools / providers / hooks have no symmetric teardown
event. This is a known limitation — do not rely on a shutdown hook.

## 6. Testing

The scaffolder drops a starter test at `tests/test_<module_name>.py`
that checks `register` is callable without importing OpenComputer at
all. That's the pattern: a plugin's own tests should need only
`plugin_sdk` + stdlib, so the plugin stays testable even if you're
developing it outside this repo.

For tests that need to exercise `register(api)` end-to-end, build a
minimal mock `PluginAPI`:

```python
class FakeAPI:
    def __init__(self):
        self.tools = []
        self.providers = {}
    def register_tool(self, tool):
        self.tools.append(tool)
    def register_provider(self, name, provider):
        self.providers[name] = provider
    # ... other register_* methods stubbed to taste

def test_my_plugin_registers_provider():
    from plugin import register  # the entry module
    api = FakeAPI()
    register(api)
    assert "my-provider" in api.providers
```

For deeper integration tests that exercise the real loader, import from
`opencomputer.plugins.discovery` and `opencomputer.plugins.loader`
directly — see
[`extensions/weather-example/tests/`](../extensions/weather-example/tests/)
and the Phase 12b2 tests under `tests/test_phase12b2_plugin_scaffold.py`
for working examples.

## 7. Common gotchas

Seven lessons burned into this codebase — read them before shipping:

1. **Sibling module-name collisions.** Two plugins each shipping
   `provider.py` can end up sharing the same class if you rely on bare
   `from provider import X`. The loader handles this via synthetic
   module names + cache scrubbing; your tests should too.
2. **Proxy header quirks.** Some LLM proxies forward `x-api-key`
   unchanged and fail auth. Providers that need `Authorization: Bearer`
   must strip `x-api-key` via an httpx event hook — see
   `extensions/anthropic-provider/provider.py` for the pattern.
3. **Compaction preserves tool pairs atomically.** Don't assume
   message history slicing is safe; `tool_use` + `tool_result` must
   stay paired or Anthropic will 400.
4. **`staticmethod` factories.** Lambdas stored as class attributes
   become bound methods. Wrap with `staticmethod(...)` for set-on-class
   factory patterns (see `DelegateTool._factory`).
5. **asyncio subprocess isolation.** A subprocess started in one
   `asyncio.run(...)` cannot be awaited in another. Keep spawn / check
   / kill in the same event loop invocation.
6. **SDK boundary is test-enforced.** `plugin_sdk/*` may not import from
   `opencomputer/*`. A test scans for violations and fails CI if any
   appear — don't try to sneak one past by moving imports into
   functions.
7. **`HookContext.runtime` is optional.** For backwards compat with
   hooks written before Phase 6a. New hooks should still expect it may
   be `None`, and use a guard: `if ctx.runtime and ctx.runtime.plan_mode: ...`

## 8. Installing

Once your plugin works, install it via the CLI:

```bash
# Profile-local (default) — installs to the active profile's plugin dir
opencomputer plugin install /path/to/my-plugin

# Global — shared across all profiles
opencomputer plugin install /path/to/my-plugin --global

# A specific named profile
opencomputer plugin install /path/to/my-plugin --profile research
```

Uninstall with `opencomputer plugin uninstall <id>`; locate with
`opencomputer plugin where <id>`. `install` refuses to overwrite an
existing install unless you pass `--force`.
