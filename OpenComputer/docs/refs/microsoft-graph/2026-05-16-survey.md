## 2. The three Graph operations — exact shapes

All requests carry `Authorization: Bearer {access_token}`. Base URL `https://graph.microsoft.com/v1.0`.

### 2.1 Send mail — `POST /me/sendMail`

- **Method + path:** `POST https://graph.microsoft.com/v1.0/me/sendMail`
- **Delegated scope (least privileged):** `Mail.Send`. No lower option exists; there is no read-only variant for send.
- **Request header:** `Content-Type: application/json` (JSON form) — required.
- **Request body** (JSON form — the form OC will use; MIME form is out of scope):

```json
{
  "message": {
    "subject": "Meet for lunch?",
    "body": {
      "contentType": "Text",
      "content": "The new cafeteria is open."
    },
    "toRecipients": [
      { "emailAddress": { "address": "frannis@contoso.com" } }
    ],
    "ccRecipients": [
      { "emailAddress": { "address": "danas@contoso.com" } }
    ]
  },
  "saveToSentItems": "false"
}
```

  - `message` (required): a `Message` resource.
  - `message.subject`: string.
  - `message.body.contentType`: `"Text"` or `"HTML"`.
  - `message.body.content`: string.
  - `message.toRecipients`: array; each element is `{ "emailAddress": { "address": "..." } }`. Note the **double nesting** — `emailAddress` then `address`. An optional `name` sibling of `address` is allowed.
  - `message.ccRecipients`, `message.bccRecipients`: same shape, optional.
  - `saveToSentItems` (optional, top-level — sibling of `message`, NOT inside it): boolean; default `true`. Only send it when you want `false`.
- **Success response:** **HTTP `202 Accepted`**, **empty body**. `202` means *accepted for delivery*, not *delivered* — delivery is async and subject to Exchange Online limits. OC's tool must treat `202` (not `200`) as success.
- **Error example** (malformed input): `400 Bad Request` with `{"error": {"code": "...", "message": "..."}}`.

### 2.2 List calendar events — use `GET /me/calendarView`, NOT `GET /me/events`

The plan asks "which" — the answer is **`calendarView`**.

- **`GET /me/events`** returns single-instance events and recurring-series *masters*. A recurring weekly standup shows up once, as a master with a `recurrence` rule. Useless for "what's on my calendar this week."
- **`GET /me/calendarView`** takes a time window and **expands recurrences** — you get one object per actual occurrence in the window. This is what a human means by "list my calendar." `calendarView` is the right choice for `GraphListCalendarTool`.

- **Method + path:** `GET https://graph.microsoft.com/v1.0/me/calendarView?startDateTime={start}&endDateTime={end}`
- **Required query params:**
  - `startDateTime`: ISO 8601, e.g. `2020-01-01T19:00:00-08:00`.
  - `endDateTime`: ISO 8601, e.g. `2020-01-02T19:00:00-08:00`.
  - Both are interpreted using the **timezone offset embedded in the value**. If no offset is present, they are treated as **UTC**. The `Prefer: outlook.timezone` header does NOT affect how these params are parsed (it only affects how `start`/`end` are *rendered* in the response).
- **Delegated scope:** least privileged is `Calendars.ReadBasic`; the plan specified `Calendars.Read` — `Calendars.Read` is a valid (higher-privileged) choice and returns full event bodies. **Recommendation: request `Calendars.Read`** (plan is correct) — `ReadBasic` strips `body`/`location` detail and the agent benefits from the fuller view. Document the trade-off.
- **Optional headers:** `Prefer: outlook.timezone="Pacific Standard Time"` to render times in a chosen zone; `Prefer: outlook.body-content-type="text"` to get plain-text bodies instead of HTML.
- **Response:** `200 OK`, body `{ "value": [ <event>, ... ], "@odata.nextLink": "..." }`.
- **Default page size: 10 items** (low — see §3 pagination). Raise with `$top` up to **1000**.
- **`event` object — key fields** (the ones a list tool should surface):
  - `id` — string, opaque.
  - `subject` — string.
  - `start` / `end` — each `{ "dateTime": "2017-...T...", "timeZone": "UTC" }` (a `dateTimeTimeZone` object — NOT a bare ISO string).
  - `location` — `{ "displayName": "...", ... }`.
  - `organizer` — `{ "emailAddress": { "name": "...", "address": "..." } }`.
  - `attendees` — array of `{ "type": "...", "status": {...}, "emailAddress": {...} }`.
  - `bodyPreview` — short plain-text preview string.
  - `isAllDay`, `isCancelled`, `webLink` (opens the event in OWA), `responseStatus`.
  - Caveat from the docs: `createdDateTime` / `lastModifiedDateTime` do **not** support `$select` on `calendarView` — query without `$select` to get them.

### 2.3 List OneDrive files — `GET /me/drive/root/children`

- **Root listing:** `GET https://graph.microsoft.com/v1.0/me/drive/root/children`
- **Named-folder listing (path notation):** `GET https://graph.microsoft.com/v1.0/me/drive/root:/{folder-path}:/children` — e.g. `/me/drive/root:/Documents/Reports:/children`. The `:` delimiters bracket the path segment.
- **By item id:** `GET /me/drive/items/{item-id}/children`.
- **Delegated scope:** least privileged `Files.Read` (plan correct). `Files.Read` covers the user's own drive; `Files.Read.All` would add shared/other drives — not needed for v1.
- **Response:** `200 OK`, body `{ "value": [ <driveItem>, ... ], "@odata.nextLink": "..." }`.
- **Default page size: 200 items.** `@odata.nextLink` appears when exceeded.
- **`driveItem` object — key fields:**
  - `id` — string.
  - `name` — string (file or folder name).
  - `size` — integer bytes.
  - `webUrl` — string, opens the item in OneDrive web.
  - `createdDateTime`, `lastModifiedDateTime` — ISO 8601 strings.
  - `folder` facet — present (e.g. `{ "childCount": 4 }`) **iff the item is a folder**.
  - `file` facet — present iff the item is a file; contains `mimeType` and a `hashes` block.
  - `package` facet — present for OneNote-style packages (treat like a folder for listing).
  - **Folder-vs-file discrimination:** test for presence of the `folder` (or `package`) facet, not the `file` facet — some items have neither cleanly.

## 3. Pagination — `@odata.nextLink`

Graph collection endpoints page server-side. Contract:

- The first response contains `value` (the page) and, **only if more pages exist**, an `@odata.nextLink` — an **absolute, fully-formed URL** (it already encodes the skip token / page state).
- To get the next page: issue a plain `GET` against the `@odata.nextLink` value **verbatim** — do NOT re-append query params, do NOT parse the skip token. The Authorization header still applies.
- Repeat until a response has **no** `@odata.nextLink`. That is the last page.
- `$top` sets the page size (mail/drive default 200; **calendarView default 10** — easy to under-fetch). `$top` max is 1000.
- A `paginate()` helper in `GraphClient` should: take an initial URL, loop `GET`-ing while `@odata.nextLink` is present, concatenate `value` arrays, and accept a `max_items` cap so the agent can't accidentally pull 10k events into context. (The plan's audit §4.2 already resized T3.2 M→L *specifically* to add this helper — correct call.)
- There is also `@odata.deltaLink` for change-tracking; **out of scope for M3** (read-only one-shot listing only).

## 4. Device-code OAuth flow (Microsoft identity platform v2.0)

This is the flow `oc auth login graph` will run. It is RFC 8628 with two Microsoft-specific deviations (flagged below).

### 4.1 Device authorization request

```
POST https://login.microsoftonline.com/{tenant}/oauth2/v2.0/devicecode
Content-Type: application/x-www-form-urlencoded

client_id={app-client-id}&scope={space-separated scopes}
```

- `{tenant}`: one of `common`, `organizations`, `consumers`, or a specific tenant GUID/domain. For a tool that must work with **both** personal Microsoft accounts and work/school accounts, use **`common`**. (`organizations` = work/school only; `consumers` = personal only.) Make this configurable.
- `scope`: space-separated. For M3 request: `Mail.Send Calendars.Read Files.Read offline_access openid` — note **`offline_access` is mandatory to get a refresh token** (see 4.4), and `openid` to get an `id_token` (lets us show *which account* logged in).

### 4.2 Device authorization response (JSON)

| Field | Type | Notes |
|---|---|---|
| `device_code` | string | long; used to poll the token endpoint |
| `user_code` | string | short; the human types this at the verification URI |
| `verification_uri` | URI | the human visits this (typically `https://microsoft.com/devicelogin`) |
| `expires_in` | int | seconds until both codes expire — **default 900 (15 min)** |
| `interval` | int | seconds to wait between token polls — typically 5 |
| `message` | string | human-readable instruction string (localizable via `?mkt=xx-XX`) |

**DEVIATION 1:** Microsoft **does NOT return `verification_uri_complete`** — it explicitly says so in the docs, even though RFC 8628 lists it as optional. OC's `DeviceCodeResponse` dataclass (`opencomputer/auth/device_code.py`) has a *required* `verification_uri_complete` field, but `request_device_code()` already populates it via `data.get("verification_uri_complete", data["verification_uri"])` — i.e. it falls back to `verification_uri`. **So OC's existing code handles this correctly.** Just be aware the "complete" URI for Microsoft is really just the plain URI; the UI should print `user_code` separately and not imply the URI pre-fills it.

### 4.3 Token polling

```
POST https://login.microsoftonline.com/{tenant}/oauth2/v2.0/token
Content-Type: application/x-www-form-urlencoded

grant_type=urn:ietf:params:oauth:grant-type:device_code
&client_id={app-client-id}
&device_code={device_code}
```

Poll every `interval` seconds. Error codes returned **with HTTP 400** while polling:

| Error code | Meaning | Client action |
|---|---|---|
| `authorization_pending` | user hasn't finished signing in | keep polling at current interval |
| `authorization_declined` | user denied consent | **stop**, report failure |
| `bad_verification_code` | wrong `device_code` sent | bug — stop, report |
| `expired_token` | `expires_in` elapsed | **stop**, report timeout |
| `slow_down` | polling too fast | **DEVIATION 2 — see below** |

**DEVIATION 2:** The Microsoft docs *table* omits `slow_down`, but Microsoft's `/token` endpoint **does** return it (it is mandated by RFC 8628 §3.5 and Microsoft's implementation follows it). OC's `poll_for_token()` already handles `slow_down` by bumping the interval +5s. **Do not "fix" OC's code to drop `slow_down` just because the MS doc table doesn't list it** — keep it.

Cross-check vs OC's `device_code.py` error handling:
- OC stops on `expired_token`, `access_denied`, `invalid_grant`. Microsoft uses `authorization_declined` (not `access_denied`) and `bad_verification_code`. **Action:** OC's `poll_for_token` will treat `authorization_declined` / `bad_verification_code` as "unknown error → surface it" (its final `else` raises `DeviceCodeError`) — functionally correct (it stops and raises) but the message says "unexpected". Minor: M3 could pass these through cleanly, but it is not a blocker — the flow still terminates correctly.

### 4.4 Successful token response

```json
{
  "token_type": "Bearer",
  "scope": "Mail.Send Calendars.Read Files.Read",
  "expires_in": 3599,
  "access_token": "eyJ0eXAi...",
  "refresh_token": "AwABAAAA...",
  "id_token": "eyJ0eXAi..."
}
```

- `access_token` — opaque to us. **Do not parse or validate it** (MS docs explicitly warn: Microsoft tokens may not be valid JWTs and may be encrypted for consumer accounts). Just send it as `Bearer`.
- `expires_in` — **~3599s (1 hour)** typical access-token lifetime.
- `refresh_token` — **only returned if `offline_access` was in the requested scope.** Refresh tokens are long-lived (default ~90 days, sliding). Used to mint new access tokens without re-prompting.
- `id_token` — JWT, only if `openid` was requested. Contains the account identity (can extract `preferred_username` to display "logged in as ...").

### 4.5 Refreshing

Use the standard OAuth refresh grant against the same `/token` endpoint:

```
POST https://login.microsoftonline.com/{tenant}/oauth2/v2.0/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token&client_id={app-client-id}&refresh_token={refresh_token}
&scope={space-separated scopes}
```

Response is the same shape as 4.4 (a fresh `access_token`, usually a rotated `refresh_token`). The GraphClient should refresh **proactively** when the token `expires_soon()` AND **reactively** on a `401` (one retry).

### 4.6 App registration prerequisite (NOT a code task, but a hard blocker)

The device-code flow needs an Azure AD **app registration** with:
- A **client id** (public client — device-code has no client secret).
- **"Allow public client flows" toggled ON** in the registration's Authentication blade. Without this, the `/devicecode` endpoint rejects the request. This is a one-time Azure portal action by whoever owns the app.
- The delegated permissions `Mail.Send`, `Calendars.Read`, `Files.Read`, `offline_access`, `openid` added (admin consent not required for these delegated scopes on `common` — the user consents at sign-in).

**Decision for M3:** OC does not ship its own Microsoft app registration today. Two options:
1. Register a public OC client and bake its (non-secret) client id in as a default — mirrors what `auth/google_oauth.py` already does with Google's public gemini-cli client id (see that file's "Public client credentials" docstring). Override via env var.
2. Require the user to bring their own client id via config / env var.
Recommend **option 1 + override** for parity with the Google path, but flag to the plan owner that *someone has to actually create the Azure app registration* — it is a real prerequisite with no code substitute. If no registration exists at build time, M3 ships with the client id read from `OPENCOMPUTER_GRAPH_CLIENT_ID` and is inert until the user supplies one (acceptable — M3 is opt-in anyway).

## 5. Throttling — `429` + `Retry-After`

- On throttle, Graph returns **HTTP `429 Too Many Requests`** with body `{"error": {"code": "TooManyRequests", ...}}` and a **`Retry-After` response header** (integer seconds).
- **Required client behavior:** on `429`, wait exactly `Retry-After` seconds, then retry. If it `429`s again, repeat with the new `Retry-After`. Honoring `Retry-After` is the fastest recovery path (Graph keeps accruing usage while you're thottled, so blind immediate retries make it worse).
- If a `429` ever arrives **without** `Retry-After` (rare for these endpoints — docs say all the surveyed APIs provide it), fall back to **exponential backoff**.
- Writes (sendMail) are throttled more aggressively than reads. There is no fixed published number — it is tenant- and load-dependent.
- M3 GraphClient should implement a small bounded retry wrapper: on `429`, sleep `Retry-After` (cap the total wait, e.g. 2 retries / 60s max, then surface the error to the agent as a clean `ToolResult(is_error=True)`). Do **not** retry forever — the agent should be told "Graph is throttling, try later."

## 6. OC's existing plumbing — what to reuse, what to build

### 6.1 OAuth — REUSE `opencomputer/auth/device_code.py`, NOT `mcp_oauth_manager`

**Plan mismatch:** T3.3 (and the audit §4.1, §4.5) say to "review `mcp_oauth_manager.py`." **There is no file named `mcp_oauth_manager.py` in OC.** The MCP OAuth code is `opencomputer/mcp/oauth.py`. And — more importantly — **it is the wrong module to reuse.**

`opencomputer/mcp/oauth.py` is built around the **MCP Python SDK's** `OAuthClientProvider` (dynamic client registration RFC 7591, RFC 8414 discovery, browser-redirect auth-code+PKCE). It is MCP-server-shaped: keyed by MCP server name, storage at `<profile>/mcp/tokens.json`. It does **not** implement the device-code flow and is coupled to `mcp.client.auth`. Reusing it for Graph would be a poor fit.

**The right reuse is `opencomputer/auth/device_code.py`** — which the plan author appears not to have known exists. It is a clean, generic, provider-agnostic RFC 8628 device-code client, already in the tree:
- `request_device_code(device_code_url, client_id, scope)` -> `DeviceCodeResponse`.
- `poll_for_token(token_url, client_id, device_code, interval, max_wait_seconds)` -> raw token dict. **Already handles `authorization_pending`, `slow_down`, `expired_token`/`access_denied`/`invalid_grant`, timeout.**
- `to_oauth_token(provider, response)` -> `OAuthToken` for storage.
- Companion `opencomputer/auth/token_store.py`: `OAuthToken` dataclass (`provider`, `access_token`, `refresh_token`, `expires_at`, `scope`, `extra`), `save_token`/`load_token`/`delete_token`, JSON store at `~/.opencomputer/auth_tokens.json` (or `$OPENCOMPUTER_HOME/auth_tokens.json`), file mode `0600`, corrupt-JSON-tolerant. `OAuthToken.is_expired()` / `expires_soon(skew)` already exist for refresh decisions.

**Verdict on T3.3:** the plan's branch "if reusable, T3.3 shrinks to S" **is satisfied** — `device_code.py` covers ~90% of the device-code flow. What `graph_oauth.py` still has to add (the genuinely Graph-specific 10%):
1. The Microsoft endpoint URLs + tenant handling (`https://login.microsoftonline.com/{tenant}/oauth2/v2.0/devicecode` and `/token`).
2. The OC Graph client id default + `OPENCOMPUTER_GRAPH_CLIENT_ID` override (mirror `google_oauth.resolve_client_credentials`).
3. A **refresh** function — `device_code.py` has NO refresh helper (it only does the initial grant). `graph_oauth.py` must POST the `grant_type=refresh_token` request itself (small — ~20 lines, see §4.5). `google_oauth.refresh_access_token()` is the pattern to copy.
4. A `get_valid_access_token()` that loads the stored token, refreshes if `expires_soon()`, persists, returns — again, `google_oauth.get_valid_access_token()` is the exact pattern.

So T3.3 is **S-to-low-M**, and it should reuse `auth/device_code.py` + `auth/token_store.py`. It should **not** touch `mcp/oauth.py`.

**One storage note:** there are *two* `OAuthToken` classes in the tree — `opencomputer/auth/token_store.py::OAuthToken` and `opencomputer/mcp/oauth.py::OAuthToken`. They are different dataclasses. M3 must use the **`auth/token_store.py`** one. Token lands in `auth_tokens.json` under provider key `"graph"` (or `"microsoft-graph"`). The plan says "store tokens in `auth.json`" — the actual file is **`auth_tokens.json`** (per `token_store.default_store_path()`). Minor naming correction; do not create a new `auth.json`.

### 6.2 `oc auth login graph` — THIS COMMAND DOES NOT EXIST AND CANNOT BE "EXTENDED"

**Largest plan mismatch.** The plan (M3 "Done when", T3.3) assumes an `oc auth login <provider>` subcommand exists to extend. It does not. Current reality:

- **`oc auth`** (`opencomputer/cli_auth.py`) is a **credential-pool manager** — subcommands `list` / `add` / `remove` / `reset` operating on `config.yaml`'s `credential_pools:` key, plus a bare-invocation env-var status table. There is **no `login` subcommand** and no OAuth anywhere in it.
- **`oc login <provider>`** (`opencomputer/cli_login.py`, top-level command) is an **API-key capture** tool — `getpass` prompt → writes `ANTHROPIC_API_KEY`-style vars into `<profile>/.env`. **So M3 must CREATE the CLI surface, not extend one.** Options:
- (a) Add a new `oc auth login <provider>` subcommand to `cli_auth.py` (the `auth_app` Typer group). This best matches the plan's wording and is where an OAuth login belongs (alongside other auth state). The bare-`oc auth` callback already exists; adding `auth login graph` is additive.
- (b) Add `graph` handling to the top-level `oc login` — rejected: `cli_login.py` is hard-wired to the API-key-into-`.env` model; OAuth is a different mechanism and would pollute it.

**Recommendation: option (a)** — new `oc auth login graph` (and `oc auth logout graph` / `oc auth status` should reflect it). T3.3's "integrate with `oc auth login graph`" becomes "**create** `oc auth login graph`." Size impact: small, but it is *new* code, not a one-line registration. Update the plan's wording from "extend" to "create."

### 6.4 `opencomputer/tools/` — `BaseTool` + capability claims

50 built-in tools live in `opencomputer/tools/*.py`. The contract (`plugin_sdk/tool_contract.py`):

- Subclass `BaseTool` (ABC). Implement the `schema` property (returns `ToolSchema(name, description, parameters)` — `parameters` is a JSON Schema object) and `async def execute(self, call: ToolCall) -> ToolResult`.
- **`execute` must never raise** — it catches its own errors and returns `ToolResult(tool_call_id=call.id, content="Error: ...", is_error=True)`. The registry's `dispatch` has a defensive catch, but tools own their errors.
- `capability_claims: ClassVar[tuple[CapabilityClaim, ...]]` — declare a **tuple** (not list — mutable-default footgun) of `CapabilityClaim`s. Empty tuple (default) = unprivileged, no gate check.
- `parallel_safe`, `max_result_size`, `strict_mode`, `loop_safe`, `sandbox_preference`, `sandbox_backend_hint` are the other class-vars (all have safe defaults).

**`CapabilityClaim` / `ConsentTier`** (`plugin_sdk/consent.py`):

```python
class ConsentTier(IntEnum):
    IMPLICIT  = 0   # user told agent in chat — no external data read
    EXPLICIT  = 1   # user clicked "enable" for a source, revocable
    PER_ACTION = 2  # per-action prompt showing the specific data
    DELEGATED = 3   # time-windowed delegated autonomy

@dataclass(frozen=True, slots=True)
class CapabilityClaim:
    capability_id: str          # e.g. "graph.mail.send"
    tier_required: ConsentTier
    human_description: str
    data_scope: str | None = None
```

The `ConsentGate` (`opencomputer/agent/consent/gate.py`) runs **before** any PreToolUse hook — it is *the* security primitive — and resolves each claim against stored `ConsentGrant`s.

**Mapping the plan's tier choices** (M3: send-mail EXPLICIT, list-calendar/list-files IMPLICIT):

- The plan says **send-mail = `EXPLICIT`**. Reading the tier semantics: `EXPLICIT` (tier 1) = "user clicked enable for a source, revocable." `PER_ACTION` (tier 2) = "per-action prompt showing the specific data." `SystemNotifyTool` — a *non-invasive* outbound action — is `EXPLICIT`. Click/keystroke tools are `PER_ACTION`. **Sending an email is an outward, hard-to-reverse action with a named recipient — it is the textbook `PER_ACTION` case** (the user should see *this* recipient + *this* subject before it goes). The plan's `EXPLICIT` is *defensible* (it gates send-mail behind an explicit one-time "enable Graph mail send" grant) but it is **weaker than the tool's blast radius warrants.** See the pre-mortem — recommendation is to **reconsider send-mail as `PER_ACTION`**, or at minimum confirm `EXPLICIT` is a deliberate, documented choice. This is a *judgement-call mismatch*, flagged for the plan owner; not a code blocker.
- list-calendar / list-files = `IMPLICIT` (tier 0). The tier-0 doc says "no external data read" — but listing the calendar **is** an external data read. **`IMPLICIT` is arguably too low for a tool that reads the user's calendar/files from a cloud account.** However: the OAuth consent the user already gave at `oc auth login graph` (granting `Calendars.Read` / `Files.Read`) is itself an explicit, revocable grant for that data source — so the *source-level* consent is already `EXPLICIT`-equivalent at the OAuth layer. Treating the individual *list* tool as `IMPLICIT` (no extra per-call gate) is then reasonable, **provided** the OAuth login is the consent checkpoint. Recommendation: make list-calendar / list-files **`EXPLICIT`** (consistent: the capability is "read my Graph data," granted once) rather than `IMPLICIT` — small hardening, keeps the gate honest about the fact that cloud data is being read. If the plan keeps `IMPLICIT`, document that the OAuth login is the consent boundary.

**Tool registration:** built-in tools are registered in `opencomputer/cli.py::_register_builtin_tools()` (around line 617) via `registry.register(SomeTool())`. Platform-specific tools are gated there (`if sys.platform == "darwin": ...`). For M3, the three Graph tools should be registered there too — but **gated on the Graph token existing**, so they don't appear when the user hasn't logged in (mirrors how optional tools stay absent — avoids the agent reaching for a tool that will only ever return "not authenticated"). Pattern: `if graph token present in auth_tokens.json: registry.register(GraphSendMailTool()) ...`. Equivalent to the plan's "inert until `oc auth login graph`."

### 6.5 `httpx.AsyncClient` — established pattern, no new dependency

- `httpx>=0.27` is a **core** dependency in `pyproject.toml` (line 33) — **Graph needs no optional extra.** M3's GraphClient is a core integration, not an opt-in pip extra. (Contrast E2B/NeuTTS which *are* `[project.optional-dependencies]`.)
- `httpx.AsyncClient` is used in ~13 places. The cleanest reference is **`opencomputer/tools/web_fetch.py`** — `async with httpx.AsyncClient(timeout=..., headers=...) as client: resp = await client.get(url)`, with `httpx.TimeoutException` / `httpx.HTTPError` caught and turned into `ToolResult(is_error=True)`. `opencomputer/agent/anthropic_client.py` is the reference for a longer-lived client with retry. GraphClient should: construct one `httpx.AsyncClient` with the base URL + bearer header, expose `mail.send()` / `calendar.list()` / `drive.list()`, catch httpx errors, and implement the `429`/`Retry-After` retry (§5) and `401`→refresh-once (§4.5).
- Note `auth/device_code.py` uses **synchronous** `httpx.post` (the device-code poll is a blocking loop — fine for a one-time interactive `oc auth login`). The GraphClient runtime path, by contrast, runs inside the async agent loop and must be **async**. Two different httpx usages, both correct for their context. `graph_oauth.refresh` can be sync (called rarely) or async — keep it consistent with how the tool calls it; sync is simplest and matches `device_code.py`.

## 8. Numbered plan-vs-reality mismatches (build-affecting)

1. **No Hermes reference exists.** T3.1 ("Read Hermes' `tools/microsoft_graph_client.py` + `microsoft_graph_auth.py`") is built on a false premise — those files, and any Graph code, are absent from the entire Hermes source. **Action: delete T3.1**; this survey replaces it.
2. **`mcp_oauth_manager.py` does not exist.** The plan (T3.3, audit §4.1 & §4.5) names it. The real file is `opencomputer/mcp/oauth.py`, and it is the **wrong** module to reuse (MCP-SDK-shaped, browser-redirect, no device-code). **Action: T3.3 reuses `opencomputer/auth/device_code.py` + `auth/token_store.py` instead** — which the plan author seems unaware exist. Net effect: the plan's "if reusable, T3.3 shrinks" branch *fires* — but pointed at a different module than named.
3. **`oc auth login <provider>` does not exist — in any form.** `oc auth` is a credential-pool manager; `oc login` is an API-key-into-`.env` tool; `auth/google_oauth.py`'s browser flow has zero CLI callers. M3 must **create** the `oc auth login graph` command, not "integrate with" or "extend" it. T3.3's wording is wrong; size is still small but it is new code.
4. **Token file is `auth_tokens.json`, not `auth.json`.** The plan says store tokens in `auth.json`; `token_store.default_store_path()` resolves `~/.opencomputer/auth_tokens.json` (or `$OPENCOMPUTER_HOME/auth_tokens.json`). Use the existing file + store under provider key `"graph"`. Do not create a new `auth.json`.
5. **`integrations/graph/` is a new code sub-convention.** `integrations/` today holds only `langfuse/` (a bare docker-compose). The plan's `integrations/graph/client.py` is fine but is the first *code* integration — remember the `__init__.py` the plan's file list omits.
6. **Capability tiers are arguably under-set** (judgement call, not a hard bug). `EXPLICIT` for send-mail is defensible but weaker than the tool's irreversibility warrants — `PER_ACTION` better fits "show the recipient before sending." `IMPLICIT` for list-calendar/list-files contradicts the tier-0 "no external data read" definition — recommend `EXPLICIT`. Flagged for the plan owner; see pre-mortem §"consent gating."
7. **`calendarView` default page size is 10** — not 200 like mail/drive. Easy to silently under-fetch a week of events. The `paginate()` helper (already correctly added by audit §4.2) plus an explicit `$top` mitigates this — make sure `calendar_list` sets `$top` and paginates.
8. **Azure app-registration is a hard, code-less prerequisite.** Device-code needs a public-client app registration with "allow public client flows" ON. No code can substitute. M3 should resolve the client id from `OPENCOMPUTER_GRAPH_CLIENT_ID` (mirroring `google_oauth`); the feature is legitimately inert until that exists — consistent with M3 being opt-in. Call this out to the plan owner so the registration gets created in parallel with the build.
9. **M2's `e2b.py` already merged.** `opencomputer/sandbox/e2b.py` exists and `pyproject.toml` has `e2b = ["e2b>=2.21,<3"]`. Not an M3 issue, but confirms the plan's milestones are landing — M3's `BaseTool` already has the `sandbox_preference` / `sandbox_backend_hint` fields from M2. Graph tools should leave both at default (`"default"` / `None`) — network API calls don't need sandboxing.
